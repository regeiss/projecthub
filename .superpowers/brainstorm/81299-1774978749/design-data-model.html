Phase 1 Design — Section 1: Data Model

Changes to WikiPage to support dual-store content

WikiPage model — before vs after
Before
class WikiPage(models.Model):
    ...
    content = JSONField(
      default=dict
    )
    # stores {"_yjs": "hex..."}
    # or TipTap JSON
    # — mixed, opaque
After
class WikiPage(models.Model):
    ...
    content = JSONField(
      default=dict
    )
    # Always TipTap JSON now
    # readable by backend

    yjs_state = models.BinaryField(
      null=True, blank=True
    )
    # Raw Yjs binary state
    # for real-time sync only
Migration plan:
  • Add yjs_state BinaryField (nullable)
  • Existing rows with {"_yjs": hex} in content: move hex → yjs_state, set content = {}
  • WikiPageVersion: same split — add yjs_state field
Save flow — how dual-store works end-to-end
WebSocket (Yjs sync)
Consumer receives binary Yjs update
→ Relay to peers (unchanged)
→ Debounce 10s → save to yjs_state
REST save (title + content)
Frontend sends TipTap JSON on Save
→ PATCH /wiki/pages/{pk}/
→ Saved to content JSONField
On load (WebSocket connect)
Consumer sends both: content (TipTap JSON for initial render) + yjs_state binary (for Yjs reconciliation). Editor shows content immediately; Yjs catches up any in-flight peer edits.
