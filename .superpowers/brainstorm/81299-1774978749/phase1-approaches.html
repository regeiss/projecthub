Phase 1 — Implementation Approaches

Key constraint: content is stored as raw Yjs binary — backend can't read headings. Choose how to handle this.

A

Dual-store: persist TipTap JSON alongside Yjs (recommended)

When saving, the frontend sends both the Yjs binary AND the TipTap JSON. Backend stores TipTap JSON in content and Yjs in a new yjs_state field. Backend can now read headings for TOC, full-text search, and diffs.

Pros

  • Unlocks backend search (Phase 2)
  • Version diffs readable (Phase 4)
  • Clean data model

Cons

  • Migration + new DB column
  • Larger payloads on save
B

Frontend-only: parse Yjs in browser, no backend changes

TOC is extracted client-side from the live TipTap/Yjs editor state. Breadcrumbs resolved by walking up the TanStack Query cache. No backend changes needed for Phase 1.

Pros

  • No migration, no backend work
  • Fast to ship

Cons

  • TOC only works while editor is mounted
  • Doesn't enable future search or diffs
  • Breadcrumbs require N cache hits for deep pages
C

Hybrid: frontend TOC + backend ancestors endpoint

TOC is frontend-only (from live editor). A new /wiki/pages/{pk}/ancestors/ endpoint returns the ancestor chain efficiently for breadcrumbs. Defers content storage refactor to Phase 2 (when search forces it).

Pros

  • Breadcrumbs are reliable and fast
  • No forced migration now
  • Ancestors endpoint reusable for graph view

Cons

  • TOC still frontend-only
  • Content refactor just deferred, not avoided
